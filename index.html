Map的深入理解(一：概览)

Map容器概览

标准Java类库包含了Map的几种基本实现，包括HashMap,TreeMap,LinkedHashMap,WeakHashMap,ConcurrentHashMap,IdentityHashMap.他们都有同样的基本接口Map，但是行为特性各不相同，主要表现在效率，键值对的保存和呈现的次序、对象保存的周期、映射表如何在多线程程序中工作和判定“键”等价的策略等方面。

容器名称 特性
HashMap* Map基于散列表的实现（它取代了HashTable）。插入和查询“键值对”的开销是固定的。可以通过构造器设置容器和负载因子，以调整容器的性能
LinkedHashMap 类似于HashMap，但是迭代遍历时，取得“键值对”的顺序是其插入次序，或者是最近最少使用（LRU）的次序。只比HashMap慢一点；而在迭代访问时反而更快，因为它使用链表维护内部次序
TreeMap 基于红黑树的实现。查看“键”或“键值对”时，它们会被排序（次序由Comparable或Comparator决定）。TreeMap的特点在于，所得到的结果是经过排序的。TreeMap是唯一带有subMap()方法的Map，它可以返回一个子树
WeakHashMap 弱键（weak key）映射，允许释放映射所指向的对象；这是为解决某类特殊问题而设计的。如果映射之外没有引用指向某个“键”，则此“键”可以被垃圾回收器回收
ConcurrentHashMap 一种线程安全的Map，它不涉及同步加锁。
IdentityHashMap 使用==代替equals()对“键”进行比较的散列映射。专为解决特殊问题而设计

Map中使用的键值对进行key-value映射，任何key都必须具有一个equals()方法；如果key被用于散列Map，那么这个key必须要具有恰当的hashCode()方法;如果key被用于treeMap，那它还需要实现Comparable接口

总结：
使用自己的类作为HashMap的键时，覆写hashCode()时需要同时覆写equals()方法，其中equals()必须满足下列5个条件：

  1. 自反性。对任意x, x.equals(x) 一定返回true
  2. 对称性。对于任意的x,y，如果y.equals(x)返回true,则x.equals(y)也返回true
  3. 传递性。对于任意的x,y,z, 如果有x.equals(y)返回true，y.equals(z)返回true，则x.equals(z)一定返回true
  4. 一致性。对于任意x和y，如果对象中用于等价比较的信息没有改变，那么无论调用多少次x.equals(y)，返回的结果应该保持一致而不发生变化。
  5. 对于任意不是null的x，x.equals(null)一定返回false

如何编写合理的hashCode()

总体原则：

  1. 无论何时，对同一个对象调用hashCode()都应该生成同样的值。如果进行put时添加进HashMap时产生一个hashCode()值，而用get()取出时却产生另一个hashCode()值，那么就无法重新获取该对象了。因此，若hashCode()方法依赖于对象中易变的数据，我们就要当心了，因为此数据发生变化时hashCode()就会生成一个不同的散列码，相当于产生了不同的key。
  2. hashCode()也不应该依赖于具有唯一性的对象信息，特别是使用this的值，它会导致很糟糕的hashCode()。
  3. hashCode()必须实用，并且必须有意义，散列码不必独一无二，然后生成的速度要尽可能快，通过hashCode()和equals()方法必须完全确定对象身份
  4. 生成的hashCode应该分布均匀
  5. 检查hashCode()确保相同对象有相同的值

在Effective Java Programming Language Guide 一书中给出了合理hashCode()的设计指导。

类型f 合理的hash值c
boolean c = (f ? 0 : 1) 在java源码中其实为c = (f ? 1231 : 1237)
byte,char,short,int c = (int)f

double long l = Double.doubleToLongBits(f); c = (int)(l^(l>>>32))

Spring的DI原理详解

DI（Dependency injection）

DI：依赖注入，实现控制反转（IoC）的一种方式。控制反转即获取依赖的资源的方式反转了，使代码模块的耦合关系变弱。

在详细了解Spring的DI之前，我们需要带着以下几个问题来学习：

  1. 什么是控制反转
  2. 为什么控制反转能降低耦合度
  3. 依赖注入是如何实现控制反转
  4. Spring如何实现依赖注入

依赖倒置

传统软件模块布局模式的依赖关系


传统的软件结构中，次层模块直接被高层模块调用和使用，这种设计结构会增加体系结构的耦合程度，高层模块直接依赖于比它低的模块来获取一些功能，会导致高层模块的复用能力被次层模块束缚。
正是由于这种缺陷，所以出现了依赖倒置模式用以避免高度耦合，同时增加高层模块的可复用能力。

依赖倒置模式的依赖关系


由上图可以看出，相比传统的依赖模式，依赖倒置模式增加了一个抽象层来解耦传统结构模式高层对次层（下文的所有“次层”代表比高层要低的层次，因为说次层比较拗口，因此下文统一说为“低层”）的直接依赖关系，该模式中，高层和低层直接依赖与它们之间的抽象层，抽象层（图中为高层接口）描述了高层模块的行为，低层模块实现该高层接口。

DI（依赖注入）

上个章节介绍了依赖倒置的概念，也正是基于这个概念和倒置模式，才生出了我们要讨论的依赖注入。
依赖：是一个对象中将要使用到的另一个对象，如UserService中的DbDriver对象，对于UserService来说，DbDriver是它的一个依赖对象（dependency），依赖于这个对象完成searchUsers操作，而UserService可以看作为需依赖对象（dependent）
注入：将依赖对象传递到需依赖对象中，而注入方式一般有三种：

  • 构造器注入
  • setter注入
  • 接口注入（用的比较少）
  • 注解注入（spring提供的，实际由上述注入方式达到注解注入）

接口注入

接口注入的方式和构造器注入和setter注入很相似但是操作更麻烦。实际上因为依赖是往往就是一个类的成员变量，因此基本和成员变量设置值的方式基本是一致的，其实很简单。个人认为重点不是在注入这个过程，上述的精华在于倒置原则的实现，使得代码层次关系解耦，使得依赖对象和需要依赖的对象不是紧密联系，然后再通过注入的形式使具体需要的依赖推迟到具体需要时再关联起来。控制反转，其实就是对象如何获得它依赖对象的引用，反转是责任的反转。

DI优缺点

优点

  • 依赖注入允许用户进行更灵活的配置，并且理论上只需对用户代码做更改即可满足需求
  • 依赖注入这种方式可以用来将系统的一些配置写在配置文件中，这样可以通过配置文件而不用进行系统的重编译，达到功能的修改
  • 更加便于单元测试
  • 通过依赖注入，使得用户代码不需要关心它需要使用的服务具体的实现细节

缺点

  • 增加了客户代码的复杂度，因为依赖的注入需要客户代码来实现。（其实很正常，有一方的活少了，总有一方的活会躲起来，毕竟这些任务都是必须的，只是通过依赖注入可以减少没必要的修改）
  • 会增加阅读的难度。因为你看到的代码，它的具体执行过程还跟它真正注入的依赖有关联，不同的依赖，执行的任务可能是不一样的，没有以前直观。但是我认为这个不是很大的缺点，规范的代码，看起来还是和看诗一样的。
  • 会倒置对依赖注入框架的过度依赖。

异常基本常识

“异常”即发生了与预期不符的错误，但是在当前环境下无法获得必要的信息来解决该问题。从而产生并抛出异常到上一级环境进行处理。
例子：除法操作，当输入一个意外的值，当前的除法操作并不能很清楚这个值是什么含义，也不知道该如何处理这个意外的值，因此会打断当前执行顺序，产生一个异常到上一级。

Java标准异常

Throwable是所有异常的基类，在Java中表示任何可以被抛出的异常类。而Throwable分为：Error类和Exception类。前者表示编译时和系统错误次异常通常不需要程序员关心，而后者是用户方法和运行时异常抛出的异常类，是程序员需要关心的。

异常的创建及其参数

跟使用java的其他对象一样，通过new在堆上创建异常对象，这个过程伴随着存储空间的分配和构造器的调用。Java能抛出任意类型的Throwable对象，因为该对象是所有异常的根类。
标准异常类都会有两个构造函数：1.默认构造函数. 2.接受字符串为参数的构造函数

在上述例子中 new关键字创建异常对象后，此对象的引用将传给throw（有点像return）

异常捕获与处理

  1. try
  2. catch

一般情况下如果方法内部出现异常，或者方法内部抛出异常，那么该方法将结束运行。如果不希望此方法就此结束，就需要将运行代码放在try块中

当在try块中发生异常时就需要在异常处理程序中进行处理，而该异常处理程序紧跟在try块后面，用catch表示

当try块发生异常时，就会将所抛出的异常到catch块中进行匹配，寻找到第一个符合的catch块，如果异常是Type2，那么就会被第2个catch捕获。（假设Type1与Type2之间没有继承关系，如果Type1是Type2的基类，那么会被第一个catch块捕获，而不会被第二个catch块捕获，所以在异常继承关系中，子层异常捕获应该置于基类异常捕获前面）

try-catch和switch-case的区别

在switch-case的所有case最后都需要跟一个break，以避免执行后面的case子句继续被执行。

终止和和恢复

终止，即程序执行发错错误，无法补救并抛出异常，程序不在继续执行发生异常的代码。
恢复，即通过异常处理程序，修正错误。但是基本上异常出来后很少能恢复。最大的功能异常是便于发生错误时定位错误。并修改代码

自定义创建异常

Java内建的异常不可能覆盖所有层面，所以我们可以自己定义一些异常来表示程序中的特定问题。（Java内建异常很多，很可能并不需要自己定义异常，而使用内建异常就够了，但是有时候实在是很难找）

异常发生时，资源清理

使用finally块进行清理

finally块在try-catch块中一定会被执行。适用于无论try块中异常是否发生都需要执行一些方法时进行使用。finally块可以做一些不包括内存回收工作（因为内存回收工作由Java垃圾回收机制负责）的其他资源清理工作。
例如成功执行文件打开操作后，运行写文件操作时若发生错误，则可以在finally块中关闭文件。

构造器发生异常时

在异常发生时，我们都需要询问“资源被正确清理了吗？”。 大多情况下是安全的，但是在构造器里发生异常时，就需要格外注意。即便使用finally块也很难做到清理，因为构造器发生异常时，该对象有部分还没被正确创建，也就无法在finally块中对该对象的资源进行清理操作。
对于需要清理资源的对象，其构造器异常可以分为两种情况：
1.构造器不允许发生失败
2.构造器允许发生失败。
针对这两种情况有不同的处理方法。

对情况1，我们需要在构造器执行后代码，加上try-catch块，对对象成功创建后，若执行对象上的其他操作发生异常进行资源的及时清理。（因为对象已经成功创建，所以可以对象不是null，因此可以清理时可以调用到对象）
对情况2，如果继续使用情况1的方法会发生这样的错误，就是对象还没有成功创建，但是却在后续的catch块中调用对象的清理方法，这个操作就又是一个异常。因此需要在创建
对象这个方法上也加上try-catch，编程嵌套的try-catch

异常的限制

异常限制点：

  1. 派生类构造器不能捕获基类构造器抛出的异常
  2. 派生类覆盖后的方法，可以抛出基类方法异常说明中的异常的派生异常（有点拗口）
  3. 派生类覆盖后的方法可以不抛出基类异常说明中的异常（即便基类方法中含有异常说明，此处也说明了，某个特定方法的“异常说明接口”在继承和覆盖的过程中“变小了”），但是派生类覆盖后的方法不能抛>出基类不含有的异常说明
  4. 异常限制对派生类的构造函数比较特殊，派生类的构造函数可以抛出任何异常，而不理会基类构造函数所抛出的异常，但是必须抛出基类构造器的异常说明中的异常
  5. 方法类型是由方法名和参数类型组成的，因此不能用异常说明来重载方法。
  6. 若代码中处理的对象是派生类，那么编译器会强制要求捕获派生类中抛出的所有异常;若向上转型成基类，那么编译器就就会要求你捕获基类的所有异常。

总结

  • 在知道如何处理异常时才捕获异常
  • 解决问题并重新调用产生异常的地方
  • 进行少量修补，并绕过异常发生的地方继续执行
  • 用别的数据进行计算，代替预计会返回的值
  • 在当前环境下能做的事情尽量做完并将相同或不同的异常交给上层处理
  • 终止程序
  • 进行代码简化
