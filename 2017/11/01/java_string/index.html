Java class文件及其结构

首先我们都知道Java文件会被编译成.class文件，经过装载，被JVM解释执行。Class文件中除了有类的版本，字段，方法，接口等描述信息外，还有一项信息是常量池，用于存放编译器生成的各种字面量和符号引用，这部分内容将在类加载后进入方法区的运行时常量池中存放。

因此实际上“hello World”常量是存放在class文件的常量池中。
对于上面的String初始化工作：

  1. 编译期间，“hello World” 字面量存放在Class文件的常量池
  2. 创建char数组分配内存，拷贝“hello World”到char数组

这个时候实际上就有两个字符串，一个是常量池中的“hello World” 另一个是我们new出来的String对象，它的值为“hello World”。
其实在Java中我们称常量池中的字符串字面量为“拘留字符串”

为什么呢？
1.在JDK1.6中，intern()方法会把首次遇到的字符串复制到方法区的常量池中，返回的是方法区常量池中的字符串实例引用，而由StringBuilder创建的字符串实例是分配在java堆中，所以肯定不是同一个引用，故为false
2.在JDK1.7后，intern()方法不会再复制实例，只会在常量池中保存首次出现的实例的引用，所以s1.intern() == s1 是为true
而当String s2 = new StringBuilder( ).append( “hello “ ).append( “world” ).toString();此时的s2是堆中又一个实例化的对象引用，s2.intern()是第一个实例化的引用所以不会相等
如果你试试System.out.println(s2.intern() == s1.intern());结果就是为true

StringBuilder类结构

我们应该注意到StringBuilder的默认长度为16或者”16+目标字符串长度”。
再来看其中的append方法中当容量不够时申请的空间长度

即：
1.扩增为原来长度的2倍+2
2.其长度可以超过最大MAX_ARRAY_SIZE，但是不能超过Int最大值

为什么默认大小是16，不是32或者64，为什么容量不够时扩展为原来大小的2倍+2
个人认为是基于效率的考虑。
假设默认大小为0，那么一旦进行append操作就需要进行容量扩展，如果每次容量扩展只是增加到实际大小。那么下一次append必然还需要扩展大小，这样就会导致每次append操作都要进行容量的扩增，将旧数据拷贝的新数组这个过程，而这个过程是非常损耗性能的。
至于为什么是16不是32或64，估计就是java人员认为的一个比较合适的初始值吧。
而对于每次扩展为原来2倍的做法实际上和网络通信中的“拥塞控制”中对于滑动窗口的“指数增长”的做法有异曲同工之妙

我们在来看一下StringBuffer

StringBuffer类方法和StringBuilder基本一致，差别在于将每个方法名前加了“synchronized”同步控制，因此StringBuffer是线程安全的
